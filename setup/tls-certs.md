# TLS certs

The firewall presents one cert that covers both hostnames. We use a Let's Encrypt SAN cert with `vpn.rooez.com` and `trading.rooez.com` on the same certificate. Let's Encrypt verifies that you control the domain through a DNS-01 challenge — they ask Cloudflare for a specific TXT record, certbot adds it, Let's Encrypt sees it, and issues the cert.

This means you do not need a public IP or a webserver yet. The cert is generated entirely off-band.

## Before you run

You need:

- A Cloudflare API token with DNS edit on `rooez.com`. See [dns-config.md](dns-config.md).
- Both A records already created (the placeholder IP from the DNS step is fine).
- certbot installed locally (`brew install certbot`).

## Use staging first

Let's Encrypt rate-limits production cert issuance to 5 per week per registered domain. While you are confirming the chain works, use the staging endpoint, which has much higher limits but produces a cert your browser will not trust.

```bash
export CF_API_TOKEN="..."
export EMAIL="dev@mariojruiz.com"

STAGING=1 scripts/generate-certs.sh
```

Look at the output. If you see a fullchain.pem and privkey.pem under `./certs/config/live/ravpn-workshop/`, the chain is working.

## Now generate the real cert

Drop the `STAGING=1` and run again:

```bash
scripts/generate-certs.sh
```

This produces the real cert, valid for 90 days, signed by a CA your browser trusts.

## Verify

Look at the SANs:

```bash
openssl x509 -in ./certs/config/live/ravpn-workshop/fullchain.pem -noout -text | grep DNS
```

Expected:

```
DNS:trading.rooez.com, DNS:vpn.rooez.com
```

Look at the expiry:

```bash
openssl x509 -in ./certs/config/live/ravpn-workshop/fullchain.pem -noout -enddate
```

Expected: a date roughly 90 days from now.

## Where the cert goes

Three places:

- cdFMC binds it to the **RAVPN connection profile**. This is what the Secure Client sees.
- cdFMC binds it to the **ZTAA Application Group**. This is what the user's browser sees during the SAML flow.
- The trading app VM uses a **separate self-signed cert** for the connection between FTDv and the app. That one is generated by the deploy script and only needs to satisfy the firewall, not a browser.

## Renewal

The cert lasts 90 days. For a one-day workshop this is fine. If the demo runs longer than expected, just re-run `generate-certs.sh` to renew.

## A note on rate limits

The staging endpoint exists for a reason. Burning through 5 production cert attempts in a row because the chain is misconfigured is a real risk. If staging works, prod will work; if staging fails, fix it there first.
